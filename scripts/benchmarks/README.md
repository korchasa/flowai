# Agent Benchmarking System

Система бенчмарков предназначена для объективной оценки качества работы AI-агентов и навыков (skills) в среде AssistFlow. Она эмулирует поведение Cursor и проверяет не только текст ответа, но и реальные изменения в файловой системе.

## Архитектура и Изоляция

### Песочница (Sandbox)
Каждый запуск сценария происходит в изолированной временной директории (`scripts/benchmarks/work/<scenario-id>/sandbox`).
- Перед запуском директория полностью очищается.
- В песочницу копируются файлы из папки `fixture/` соответствующего сценария.
- Все команды агента (bash-скрипты) выполняются внутри этой директории.

### Runner (Оркестратор)
Управляет жизненным циклом теста:
1. **Setup**: Создание песочницы, копирование фикстур, выполнение `scenario.setup()`.
2. **Context Assembly**: Сборка системного промпта и подготовка `AGENTS.md`.
3. **Execution Loop**: Итеративное взаимодействие с LLM (до 10 шагов по умолчанию).
4. **Evidence Collection**: Сбор логов, `git status`, `git diff` и состояния файлов после работы агента.
5. **Judging**: Оценка результатов с помощью LLM-судьи.

## Сборка контекста

Runner формирует системный промпт, максимально приближенный к реальности в Cursor:

1. **Системные инструкции**: Используется шаблон из `lib/prompts.ts`, включающий `user_info` (OS, Shell, Date). Шаблон содержит следующие плейсхолдеры:
   - `{{AGENTS}}`: Сюда подставляется содержимое файла `AGENTS.md`. Это ключевой элемент, задающий общие правила поведения и контекст проекта.
   - `{{SKILL}}`: Сюда подставляется содержимое `.md` файла тестируемого навыка (из `targetAgentPath`). Это определяет конкретную логику и шаги выполнения задачи.
2. **AGENTS.md**: 
   - **Обязателен** для каждого сценария.
   - Ищется в `fixture/AGENTS.md` или передается явно в `agentsMarkdown`.
   - Подставляется в секцию `<rules>` системного промпта.
   - Копируется в корень песочницы, чтобы агент мог его прочитать.
3. **Skill/Agent Definition**: Содержимое `.md` файла навыка (например, `af-plan/SKILL.md`) подставляется в конец системного промпта.

## Обработка ответов агента

Агенты в режиме бенчмарка работают в режиме эмуляции инструментов:
- Они **не используют** реальные MCP инструменты или `todo_write`.
- Вместо этого они выводят команды в блоках кода ` ```bash `.
- Runner перехватывает эти блоки, записывает их во временные `.sh` файлы внутри песочницы и исполняет.
- Результат выполнения (stdout/stderr) возвращается агенту как `Command Output`.

## Оценка результатов (Judging)

Оценка проводится LLM-судьей на основе:
1. **User Query**: Изначальный запрос пользователя.
2. **Full Log**: Вся история переписки агента и выводы команд.
3. **Evidence**: Итоговое состояние git, список выполненных команд и изменения в файлах.
4. **Checklist**: Список критериев из `mod.ts` сценария.

Критерии бывают:
- **Critical**: Если проверка не пройдена, весь сценарий считается проваленным (`FAILED`).
- **Non-critical**: Влияют на итоговый счет, но не приводят к провалу.

## Как создать новый бенчмарк

1. Создайте директорию в `scenarios/af-<skill>/<name>`.
2. Создайте `fixture/AGENTS.md` с описанием проекта для этого теста.
3. Создайте `mod.ts`, экспортирующий объект `BenchmarkScenario`.
4. Определите `checklist` с четкими критериями проверки.
5. Запустите для проверки:
   ```bash
   deno task bench -f <scenario-id>
   ```

## Отладка

После каждого запуска в папке `work/<scenario-id>/` создаются:
- `sandbox/`: Итоговое состояние файлов (можно зайти и проверить руками).
- `trace.md`: Полный лог выполнения, включая системные промпты, ответы LLM, выводы команд и аргументацию судьи.
