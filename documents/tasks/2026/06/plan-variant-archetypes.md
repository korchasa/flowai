# Plan Mandates Three Solution Archetypes for Non-Obvious Tasks

## Goal

When `plan` analyzes a non-obvious task, the generated solution-variant set MUST cover three distinct archetypes by default — a quick fix, an architecturally-correct option, and the best long-term option — so the user always sees the short/correct/strategic spectrum instead of an arbitrary "2+ variants" set. Raises decision quality and makes the trade-off axis (speed vs. correctness vs. longevity) explicit and consistent across `plan` and every consumer that inherits the atom.

## Overview

### Context

`plan` Step 4 ("Strategic Analysis & Variant Selection") currently mandates only "**2+ distinct** implementation approaches for non-trivial tasks" and keeps an explicit single-variant exception for obvious tasks. The canonical phrasing of the policy lives in two places that must agree:

- `framework/atoms/plan.md` Step 4 (the text the model reads directly).
- `framework/core/assets/AGENTS.template.md` → `## Planning Rules` → `Variant Analysis` bullet (referenced by `plan`/`ship` via "following `Variant Analysis` from AGENTS.md").

`ship` inherits the variant policy because its SKILL.md is generated from the `plan` atom (`framework/composites.yaml` → ship includes `atom: plan`). `epic` does phase decomposition, not solution-variant analysis, so it is out of scope.

User decisions (this session):
1. The ≥3-archetype mandate applies **only to non-obvious tasks**. The existing exception "obvious task → exactly one variant" stays. The `plan-variants-obvious` acceptance scenario MUST keep passing unchanged.
2. Archetype definitions (agreed):
   - **Quick fix** — minimal change within current scope; may incur tech debt.
   - **Architecturally-correct** — correct design within the task's current constraints/scope.
   - **Best long-term** — strategic; may exceed current scope (refactor/investment); optimizes maintainability over the horizon.
3. **Separation of concerns (final architecture):**
   - `AGENTS.md` → `Variant Analysis` is an **abstract, domain-agnostic comparison format** for ANY set of alternatives (approaches, tools, libraries, designs, vendors, data models, …). It defines only HOW to present/compare options (per-option Pros/Cons/Risks/Best-for + cross-option Trade-offs). It MUST NOT mention plans, archetypes, or a fixed count.
   - `plan` (atom, Step 4) owns WHICH options to generate for **plan-execution** choices: the three archetypes for non-obvious tasks; it references the AGENTS.md format for presentation.
   - `ship` inherits via composite regeneration from the `plan` atom. `epic` (phase decomposition, not solution variants) is out of scope.
4. Implementation decisions locked this session: keep the rule **name** `Variant Analysis` (no rename — referenced by the Step-4 rule list and acceptance fixtures); generalize only the bullet **body**; keep the bullet under `## Planning Rules` (Step 4 references that section by name). Physically relocating the abstract canon to a general (non-planning) section is a deferred follow-up.

### Error / edge handling

- Archetype overlap (architecturally-correct ≈ best long-term): atom text instructs that when two archetypes collapse into one option for a given task, the agent states so explicitly and still surfaces the distinct third — never silently drops to two without noting the collapse.
- Label tolerance: the benchmark checks intent (minimal / correct-now / strategic), not exact headings, so wording variation does not cause false failures.
