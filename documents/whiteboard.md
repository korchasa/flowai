# Перенос бенчмарков в папки скиллов

## Goal

Упростить поддержку и разработку скиллов, объединив их код, документацию и тесты (бенчмарки) в одном месте. Это позволит разработчикам видеть примеры использования и проверять работоспособность скилла, не переключаясь между далекими папками.

## Overview

### Context

1.  **Текущая проблема**: Бенчмарки (сценарии и фикстуры) находятся в `scripts/benchmarks/scenarios/`, в то время как сами скиллы — в `catalog/skills/`. Это нарушает принцип локальности и затрудняет поиск тестов для конкретного скилла.
2.  **Рабочие папки**: При запуске бенчмарков создаются временные папки в `./benchmarks`. Пользователь хочет, чтобы они также были локализованы (хотя это требует уточнения, так как рабочие папки обычно игнорируются git и могут замусорить папки скиллов).
3.  **Видимость для AI**: Cursor по умолчанию может игнорировать папки с тестами или временными данными, поэтому нужно настроить `.cursorignore`.
4.  **Технический долг**: В `runner.ts` захардкожены пути к сценариям и фикстурам, которые нужно будет обновить.
5.  **Релевантные файлы**:
    *   `catalog/skills/` — целевое местоположение.
    *   `scripts/benchmarks/scenarios/` — текущее местоположение.
    *   `scripts/task-bench.ts` — точка входа для запуска.
    *   `scripts/benchmarks/lib/runner.ts` — логика поиска фикстур.
    *   `.cursorignore` — настройки видимости для Cursor.

### Current State

*   Сценарии: `scripts/benchmarks/scenarios/<skill_name>/<scenario_name>/mod.ts`
*   Фикстуры: `scripts/benchmarks/scenarios/<skill_name>/<scenario_name>/fixture/`
*   Рабочие папки: `./benchmarks/<scenario_id>/`
*   Импорты в `task-bench.ts` жестко прописаны.
*   `runner.ts` содержит эвристику для поиска фикстур по ID сценария.

### Constraints

*   Использовать Deno.
*   Соблюдать структуру GODS.
*   Не использовать `SwitchMode`.
*   Обновить `.cursorignore`.

## Текущая ситуация и исследования (01.02.2026)

### Выясненные факты о cursor-agent
1. **Формат JSON**: При использовании `--output-format json`, агент выводит **один JSON-объект** строго перед завершением процесса.
2. **Отсутствие истории**: В JSON-ответе **нет** поля `messages`. История хранится на стороне сервера сессий.
3. **Поля ответа**: Объект содержит `result` (текст ответа), `session_id`, `subtype` (всегда `"success"`, даже если требуется ввод пользователя) и метаданные.
4. **Интерактивность**: Для эмуляции диалога `SpawnedAgent` должен сам накапливать историю, добавляя промпты пользователя перед запуском и ответы ассистента после.

### Проблемы и Root Causes
1. **Рекурсия при копировании**: Из-за переноса бенчмарков в `catalog/`, скрипт подготовки sandbox пытался бесконечно копировать `catalog/` внутрь `catalog/.../benchmarks/runs/`.
   - *Решение*: Добавлены исключения `benchmarks`, `runs`, `tmp` в `copyRecursive`.
2. **Преждевременное завершение диалога**: `SpawnedAgent` делал `break` сразу после получения первого `result`, не давая эмулятору пользователя шанса ответить.
   - *Решение*: Рефакторинг цикла `run()` — теперь выход происходит только если эмулятор вернул `null` или достигнут лимит шагов.
3. **Сломанные тесты**:
   - `spawned_agent.test.ts`: Тесты ожидали `messages` в JSON и имели устаревшие проверки количества сообщений.
   - `integration.test.ts`: Тест создания файла падал, так как судья не видел содержимого файла в `evidence`.
   - `task-test.test.ts`: Не учитывал добавленный флаг `--coverage`.

### Исправленные тесты (01.02.2026)

**Проблема**: 3 unit-теста в `spawned_agent.test.ts` падали из-за неверных ожиданий.

**Root Cause**: Тесты были написаны с предположениями, не соответствующими документированному поведению `SpawnedAgent.run()`:
- `getResponse()` вызывается после КАЖДОГО шага, а не только при "запросе ввода"
- Без `userEmulator` цикл выходит после первого шага (задокументировано)
- Если `prompt` не передан, он не появляется в `messages`

**Исправления**:
1. **Lifecycle with Resume** (строка 94): `inputCalled === 1` → `inputCalled === 2`
2. **Max Steps** (строки 139-173): добавлен mock-эмулятор для управления итерациями
3. **Message Accumulation** (строка 271): добавлен `prompt: "initial prompt"`

**Результат**: `deno task check` — 23 passed | 0 failed
