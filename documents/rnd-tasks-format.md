Ниже представлен эталонный формат задачи для AI-IDE, оптимизированный для минимизации галлюцинаций и максимальной точности генерации кода на основе результатов исследований.

# НАЗВАНИЕ ЗАДАЧИ

## 1. Objective (Цель)

**Описание:** Декларативное описание желаемого конечного состояния.
**Зачем это нужно:** Дает модели четкий ориентир для планирования действий в автономном режиме (Agent Mode).
**Пример:** `Реализовать систему двухфакторной аутентификации (2FA) через TOTP.`

## 2. Context (Контекст и привязки)

**Описание:** Прямые ссылки на файлы, документацию и внешние ресурсы через @-упоминания.
**Зачем это нужно:** AI-IDE (Cursor, Windsurf) используют эти ссылки для индексации кода. Markdown-структура здесь наиболее токен-эффективна (на 34-38% эффективнее JSON).
**Пример:**

* Базовая логика: @src/auth/auth.service.ts
* Схема БД: @prisma/schema.prisma
* Документация API: @docs/api-specs.md

## 4. Tech Stack (Технологический стек)

**Описание:** Список используемых библиотек и их версий.
**Зачем это нужно:** Предотвращает использование устаревших методов или библиотек, которые модель может предложить по умолчанию.
**Пример:** `Next.js 15 (App Router), TypeScript 5.6, Tailwind CSS, Prisma ORM.`

## 5. Strategy / Implementation Plan (Поэтапный план)

**Описание:** Список фаз реализации. Рекомендуется использовать технику Chain-of-Thought (CoT).
**Зачем это нужно:** Разбиение сложной задачи на подзадачи повышает точность решения на 15-28%. Для сложных правок используйте требование "Wait for approval" после каждой фазы.
**Пример:**

* Phase 1: Добавить поле `2fa_secret` в таблицу User.
* Phase 2: Создать эндпоинт `/auth/2fa/generate` для создания QR-кода.
* Phase 3: *STOP* — Предоставь план миграции БД перед выполнением.

## 6. Constraints & Anti-patterns (Ограничения)

**Описание:** Список запретов (что НЕ делать).
**Зачем это нужно:** Предотвращает типичные ошибки, такие как удаление комментариев, внедрение лишних зависимостей или сокращение кода (placeholders).
**Пример:**

* Не использовать внешние библиотеки для генерации QR; только @lib/qr-generator.ts.
* Не изменять сигнатуру метода `validateSession`.
* Выводить ПОЛНЫЙ код файлов без сокращений `//... существующий код`.

## 7. Definition of Done (DoD - Критерии готовности)

**Описание:** Чек-лист для проверки результата.
**Зачем это нужно:** Превращает субъективную оценку в объективно проверяемый результат, минимизируя галлюцинации.
**Пример:**

* [ ] Все Unit-тесты для сервиса аутентификации проходят успешно.
* [ ] Добавлены JSDoc-комментарии для всех новых функций.
* [ ] Код соответствует правилам линтинга проекта.

### Обоснование выбора формата:

1. **Синтаксис:** Использование Markdown в качестве основного контейнера обеспечивает лучшую токен-эффективность и читаемость. Для вложенных структур данных внутри блоков кода (например, схем БД) рекомендуется использовать YAML, так как он показал наивысшую точность (62.1%) в задачах иерархической навигации.


2. **Изоляция блоков:** Использование заголовков (`#`, `##`) помогает модели четко разделять инструкции, контекст и ограничения, что критично для моделей уровня Claude 3.5/3.7 и GPT-5.


3. **Методология:** Внедрение этапа "Strategy" задействует возможности рассуждения (reasoning), что позволяет поймать архитектурные ошибки до написания кода.
