# Agent Benchmarking System Requirements (SRS)

## 1. Introduction

### 1.1 Purpose

This document defines the requirements for a robust, evidence-based benchmarking system designed to evaluate Autonomous AI Agents. The system aims to verify agent capabilities, reliability, and efficiency in a safe, isolated environment.

### 1.2 Scope

The system covers the execution lifecycle of agent scenarios, from environment setup to result evaluation. It targets agents operating within a CLI/IDE environment (file manipulation, git operations, shell execution).

### 1.3 Benchmarking Modes

The system MUST support three primary evaluation modes:

1. **Quality Evaluation (Checklist-based)**:
   - **Goal**: Verify if an agent meets the minimum quality standards.
   - **Method**: Evaluates a single agent against a predefined checklist of criteria (Critical Errors vs Warnings).
   - **Metrics**: Pass/Fail status, Errors count, Warnings count, Steps taken, Time, Financial Cost.
   - **Use Case**: CI/CD pipelines, regression testing, daily health checks.

2. **Model Selection (Pairwise Comparison)**:
   - **Goal**: Determine which LLM/Model performs best on a specific task.
   - **Method**: **LLM-as-a-Judge Side-by-Side (SBS)**. The Judge receives the task context and the outputs (logs, artifacts) from two different models. It analyzes them concurrently and selects the winner based on reasoning, quality, and efficiency.
   - **Metrics**: Win Rate, ELO Rating.
   - **Use Case**: Choosing the default model for a new agent or feature.

3. **Version Comparison (Regression/Improvement Tracking)**:
   - **Goal**: Measure the impact of changes to the agent's prompt or logic.
   - **Method**: **LLM-as-a-Judge Side-by-Side (SBS)**. The Judge compares the execution of the current version (HEAD) against a baseline (BASE). It determines if the changes resulted in better logic or output quality, essentially treating the versions as two competing models.
   - **Metrics**: Win/Loss vs Baseline (Quality), Delta values (Cost, Time) for human review.
   - **Use Case**: optimizing prompts, refactoring agent logic.

## 2. Core Philosophy

### 2.1 Evidence-Based Verification

- **Principle**: Trust nothing the agent says; verify only what it does.
- **Implementation**: Evaluation must be based on "Side Effects" (file system state, git history, exit codes) and not on parsing the agent's natural language response.

### 2.2 Isolation & Safety

- **Principle**: Benchmarks must never impact the host system or the project repository.
- **Implementation**:
  - All tests run in ephemeral **Sandboxes** (`work/<id>/sandbox`).
  - Sandboxes are cleaned before every run.
  - Agents are restricted to the sandbox directory (chroot-like behavior).

### 2.3 Determinism

- **Principle**: Tests should be reproducible.
- **Implementation**:
  - Use **Fixtures** to establish a known initial state.
  - Mock external non-deterministic inputs (Time, Network, Randomness) where possible.

## 3. Functional Requirements

### 3.1 Execution Engine (The Runner)

#### 3.1.1 Multi-turn Interaction (REPL)

- The runner MUST support an iterative loop:
  1. Agent generates a command.
  2. Runner executes the command.
  3. Runner feeds `stdout`/`stderr`/`exit_code` back to the Agent.
  4. Agent decides the next step.
- **Constraint**: The loop MUST have a configurable `max_steps` limit (e.g., 10 steps) to prevent infinite loops.

#### 3.1.2 Tool Execution

- The runner MUST intercept and execute shell commands generated by the agent.
- Supported tools:
  - `bash`/`shell`: General command execution.
  - `fs`: File reading/writing (optional, if not done via shell).

#### 3.1.3 Mocking & VCR

- The system SHOULD support mocking external tools (e.g., `gh`, `curl`) to prevent real network calls during testing.
- **Future**: Implement VCR-style recording of network interactions.

### 3.2 Evaluation (The Judge)

#### 3.2.1 LLM-as-a-Judge

- An impartial LLM (using the SOTA models) MUST evaluate the final state.
- Input to Judge:
  - User Query (Task).
  - Execution Log (Trace).
  - **Evidence** (Git diffs, File contents, Tree structure).
  - Checklist (Criteria).
- **Evaluation Stability & Quality**:
  - **Chain-of-Thought (CoT)**: The judge MUST provide reasoning before giving a final score.
  - **Self-Correction**: The judge SHOULD review its own evaluation for consistency.
  - **Few-Shot Prompting**: Provide examples of "Good" vs "Bad" outcomes in the judge's prompt.
  - **Multi-Judge Consensus**: Optionally use multiple different models as judges to reduce bias.
  - **Order Bias Mitigation**: In pairwise mode, swap the order of models and re-evaluate to ensure the judge isn't biased towards the first or second response.
- **Advanced Quality Techniques**:
  - **Adversarial Perturbations**: The system SHOULD support dynamic changes to fixtures (e.g., renaming variables, adding noise files) to test agent robustness.
  - **Efficiency-Weighted Scoring**: Evaluation SHOULD penalize redundant actions (e.g., repeated file reads, non-state-changing commands) even if the final result is correct.
  - **Evidence Auto-Discovery**: The system SHOULD automatically collect all side effects (git diffs, file changes, command logs) to provide a complete context for the judge.

#### 3.2.2 Pass@k (Statistical Significance)

- The system SHOULD support running a scenario `k` times (e.g., 5 runs).
- Success is defined as `success_rate >= threshold` (e.g., 80%).

#### 3.2.3 Cost & Efficiency Metrics

- The system MUST track:
  - **Financial Cost**: Total cost of LLM usage (input/output tokens).
  - **Steps Taken**: Number of turns in the loop.
  - **Duration**: Wall-clock time.
  - **Token Usage**: Prompt and completion tokens.
- **Constraint**: Tests can fail if they exceed a "Budget" (cost or steps), even if the functional result is correct.

### 3.2.4 Output Normalization

- **Principle**: Evaluation should focus on semantic correctness, ignoring irrelevant formatting differences or environmental artifacts.
- **Implementation**:
  - **Text Standardization**: The system SHOULD normalize line endings (e.g., to LF) and encoding (UTF-8) for all captured text evidence.
  - **Noise Reduction**: When performing strict comparisons (non-LLM), the system SHOULD support ignoring whitespace, blank lines, or specific patterns (e.g., timestamps, absolute paths) that vary between runs.
  - **Judge Context**: For LLM-based evaluation, raw outputs SHOULD be preserved, but normalized versions MAY be provided to reduce token usage and distraction.

### 3.3 Reporting (The Logger)

#### 3.3.1 Traceability

- Every run MUST generate a persistent artifact (`trace.md`).
- The trace MUST contain:
  - Full conversation history (System prompt, User messages, Agent thoughts).
  - Full execution log (Commands, Outputs).
  - Final Evidence snapshot.
  - Judge's reasoning.

## 4. Non-Functional Requirements

### 4.1 Modularity

- Scenarios MUST be defined independently of the runner logic.
- New scenarios can be added without modifying core infrastructure.

### 4.2 Portability

- The architecture SHOULD be adaptable to different languages (TS/Deno, Python, Node.js), though the reference implementation is in Deno.

### 4.3 Developer Experience (DX)

- **Scaffolding**: Provide tools to set up the infrastructure.
- **Templates**: Provide instructions for writing tests.
- **Feedback**: Console output must clearly distinguish between `PASSED`, `FAILED`, and `WARNING` states with color coding.

## 5. Data Model

### 5.1 Scenario Definition

```pseudocode
Structure BenchmarkScenario:
    id: String              // Unique identifier (e.g., "af-commit-basic")
    name: String            // Human-readable name
    targetAgentPath: Path   // Path to the agent/skill definition
    setup: Function(sandboxPath: Path) -> Promise // Fixture setup logic
    userQuery: String       // The initial task for the agent
    checklist: List<ChecklistItem> // Evaluation criteria
    maxSteps: Integer       // Optional limit on interaction turns (default: 10)
    mocks: Map<String, String> // Optional tool mocks (command -> response)
```

### 5.2 Checklist Item

```pseudocode
Structure ChecklistItem:
    id: String              // Unique identifier within the scenario
    description: String     // What the judge should verify
    critical: Boolean       // True: Failure fails the test. False: Results in a warning.
    type: Enum              // Optional: "static" (regex/grep) or "semantic" (LLM judge)
```
